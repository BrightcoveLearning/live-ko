페이지 내용

    라이브 모듈 지침 및 모범 사례

    Video Cloud Live를 사용하면 라이브 이벤트를 쉽게 설정하고 웹, iOS 및 Android 디바이스로 멀티 비트레이트 스트림을 전달할 수 있습니다. 이 항목에서는 고품질의 안정적인 라이브 스트리밍 환경을 보장하는 데 도움이 되는 일련의 모범 사례 및 권장 사항에 대해 간략하게 설명합니다.
    참고: 라이브 스트리밍 지원은 Video Cloud Pro 및 Enterprise 게시자만 이용할 수 있습니다.

    라이브 이벤트 방송을 빠르게 시작하려면 다음을 확인하십시오.

    • 단계별: 텔레스트림 와이어캐스트와 함께 라이브 모듈 사용
    • 단계별: 오픈 브로드캐스터 소프트웨어 (OBS) 와 함께 라이브 모듈 사용
    브라이트코브에서 라이브 스트리밍 비디오 전송에 사용할 수 있는 모든 옵션 목록은라이브 스트림 전송을 참조하십시오.

    인증 엔코더

    지원되는 인코더 목록은라이브 이벤트용 지원되는 인코더를참조하십시오. 다른 하드웨어 및 소프트웨어 기반 인코딩 도구는 라이브 모듈에서 작동할 수 있지만 그 사용은 지원되지 않습니다. 특정 도구에 대한 질문이 있는 경우 자세한 내용은브라이트코브 지원에문의하십시오.

    입력 스트림 가이드라인

    시청자에게 최상의 사용자 경험을 보장하는 유일한 방법은 고품질의 안정적인 입력 스트림을 제공하는 것입니다. 양호한 입력 스트림은 한 위치에서 지속적으로 사용 가능한 가장 높은 대역폭에서 최상의 비디오 품질을 제공합니다.

    입력 제한

    브라이트코브 라이브는 최고의 품질과 일관된 스트리밍 경험을 보장하기 위해 입력 스트림 설정을 다음과 같이 제한합니다.

    • 최대 용량
    • 최대 30fps, 이는 전형적입니다. 브라이트코브는 최대 60fps를 지원하지만 한도를 늘리려면브라이트코브 지원에 문의해야합니다. 60fps를 사용하는 경우 브라이트코브는 비트 전송률을 크게 높이고 일정한 프레임 속도를 사용할 것을 권장합니다.
    • 10Mbps 미만
    • 일정 비트레이트 (CBR) 는 문제 가능성을 크게 줄입니다.
    • 오디오 샘플링 속도: 44.1kHz 및 48kz는 권장되는 샘플 오디오 레이트입니다.
    • 키프레임 속도 또는 GOP (사진 그룹) 정렬:
      1. 키프레임은입력과 출력 모두에 대해항상 2초마다 발생해야 합니다. 즉, 스트림에서 2초마다 인코더에서 브라이트코브로 키프레임이 전송됩니다. 이 프로세스는 여러 가지 방법으로 정의할 수 있지만키프레임 속도가가장 일반적입니다.
      2. 다른 인코더에 의해 다른 방법으로 계산할 수 있습니다. 예:
        • 와이어캐스트는 통과하는 프레임 양을 사용하므로 30fps 비디오의 경우 설정은 60이 됩니다.
        • 엘리멘탈 엔코더는 초를 사용하므로 올바른 설정이 '2'가 됩니다.
        • 60 FPS 비디오는 이 설정이 프레임으로 계산되는 경우에만 변경됩니다. 이 경우 120 프레임마다 2 초가 됩니다.
    • [ 키프레임 정렬 ], [ GOP 동기화 ], [키프레임 정렬 ] 또는 해당 선을 따라 다른 항목이 있는 경우 키프레임이 정렬되어 있는지 확인합니다. 키프레임이 정렬되지 않으면 HLS 세그먼테이션에서 동기화 문제가 발생합니다.

    대역폭 테스트

    입력 스트림에 대한 적절한 설정에 도달하는 첫 번째 단계는 사이트에서 사용 가능한 대역폭을 결정하는 것입니다. 도움이 될 수 있는 몇 가지 도구가 있습니다.

    • Speedof.me ( http://speedof.me ) - HTTP 연결에 사용할 수 있는 총 대역폭을 결정하는 것이 좋은 첫 단계입니다. 그러나 입력 피드가 HTTP 대신 RTMP를 통해 Live 모듈로 스트리밍되므로 RTMP 연결에 사용할 수 있는 실제 대역폭은 상당히 줄어듭니다.
    • Speedtest ( http://www.speedtest.net ) - 현재 업로드 및 다운로드 속도를 결정하는 온라인 도구입니다.

    입력 대역폭

    시청자에게 최상의 사용자 경험을 보장하는 유일한 방법은 고품질의 안정적인 입력 스트림을 제공하는 것입니다. 양호한 입력 스트림은 한 위치에서 지속적으로 사용 가능한 가장 높은 대역폭에서 최상의 비디오 품질을 제공합니다.

    • 최소 입력 대역폭: 2.5 메가 비프
    • 최대 입력 대역폭: 10 메가 프스
    참고: 최상의 사용자 환경을 위해 출력 변환을 인코더 스트림의 품질에 맞추십시오.

    인코더 기능 판단

    라이브 스트림을 인코딩하여 라이브 모듈로 전송하는 데 사용되는 소프트웨어 및 하드웨어의 기능을 이해하는 것도 중요합니다. 고품질의 1080p 입력 스트림을 보내려면 많은 비트 전송률이 있을 수 있지만 하드웨어는 실시간 속도보다 빠른 속도로 인코딩 할 수 있어야합니다. 일부 인코딩 도구는 사용 중인 총 CPU 사용량 및 대역폭에 대한 정보를 표시합니다. 예를 들어, 텔레스트림 와이어캐스트는 페이지 상단에 출력 통계를 표시합니다. 이 정보는 주어진 하드웨어에서 가능한 가장 안정적이고 최고 품질의 스트림을 결정할 때 유용합니다.

    와이어캐스트에서 볼 수 있는 사항:

    • CPU는 80% 미만이어야 합니다.
    • 데이터 레이트는 대상 비트 전송률 근처에 있어야합니다.
    • FPS는 입력 스트림 설정 속도에 있어야합니다.

    이벤트 준비

    프로덕션 이벤트에 라이브 모듈을 사용하기 전에 전체 엔드 투 엔드 체인을 테스트하여 성공을 보장하는 것이 중요합니다. 테스트할 두 가지 주요 영역이 있습니다.

    • 비디오 제작 워크플로우
    • 라이브 모듈에 연결하는 데 사용되는 도구

    새로운 위치나 하드웨어 또는 소프트웨어 구성이 변경될 때마다 새로운 테스트 라운드를 수행해야 합니다.

    제작

    라이브 피드가 생성되어 인코딩 하드웨어 또는 소프트웨어로 전송되는 것은 웹캠처럼 간단하고 멀티 카메라, 편집된 라이브 브로드캐스트처럼 복잡할 수 있습니다. 두 경우 모두 들어오는 비디오 피드는 이벤트 전체에서 일관성이 있어야 합니다. 카메라 앵글이나 시스템 간에 전환하면 오디오 채널의 해상도와 수가 같은지 확인합니다.

    라이브 모듈에 연결

    라이브 모듈에 대한 연결은 프로덕션 이벤트에 사용되는 것과 동일한 인코딩 소프트웨어를 통해 전송되는 동일한 입력 라이브 피드를 사용하여 동일한 네트워크에서 테스트해야 합니다. 목표는 프로덕션 이벤트를 최대한 가깝게 모방하는 것입니다.

    이벤트의 날

    예약된 방송보다 최소 10분 전에 실시간 스트림을 시작할 계획이어야 합니다. 30 분 일찍 시작하면 문제를 조정하고 해결할 수있는 추가 시간이 제공됩니다.

    라이브 스트리밍 중 문제

    라이브 스트림을 테스트하고, 입력 스트림에 대한 이상적인 설정을 찾고, 이벤트를 위한 장소를 준비하면 오류가 발생할 가능성을 크게 줄일 수 있습니다. 그러나 발생할 수있는 문제를 항상 처리 할 준비를하는 것이 중요합니다.

    연결 해제

    네트워크 정전, 엔코더 오작동 또는 정전이 모두 Live 모듈 연결이 끊어질 수 있습니다. 라이브 이벤트에는 스트림이다시 연결될때까지 대기하는 시간 (분) 인 다시 연결 시간이 있습니다. 연결이 끊어지면 선택한 다시 연결 시간 후에 라이브 모듈은 의 연결을 끊습니다. 이벤트를 만들 때고급 옵션을확장하여 다시 연결 시간을 설정합니다.

    라이브 모듈은 새 스트림을 시작할 필요 없이 이 재연결 시간 동안 여러 개의 연결 해제 및 다시 연결을 허용합니다. 스트림이 다시 연결되면 가장 최근 프레임에서 재생이 재개됩니다. 스트림이 다시 연결되지 않고 이벤트가 종료되면 플레이어는 스트림을 찾을 수 없다는 경고를 반환합니다.

    참고: 연결을 끊었다가 다시 연결할 때 스트림 설정이 동일하게 유지되어야 합니다. 오디오 채널 수, 해상도 또는 코덱 설정을 변경하면 예기치 않은 동작이 발생합니다.

    대기 시간

    스트리밍 지연 시간 또는 시청자가 보고 있는 이벤트와 실제 라이브 이벤트 간의 시간 차이는 다음 세 곳에서 도입됩니다.

    • 현장 인코더
    • 클라우드 트랜스코딩
    • CDN 전송

    현장 인코더와 Live 모듈의 클라우드 트랜스코딩 시스템은 총 1.5초에서 5초의 대기 시간을 제공합니다. CDN에 따라 또 다른 15-90 초의 대기 시간이 발생할 수 있습니다. 일반적으로 라이브 피드의 경우 15-90초 사이의 대기 시간은 정상적인 것으로 간주되며 라이브 모듈이 실시간 이벤트를 시청하는 수백만 명의 동시 시청자에게 서비스를 제공하려면 필요합니다.

    실시간 스트림 문제 해결

    문제 해상도
    재생이 멈추고 오디오가 계속됩니다. 인코더에 대한 입력 피드를 확인하고 인코더가 제대로 연결되어 있고 비디오를 인코딩하는지 확인합니다. 연결이 끊어진 경우 다시 연결하기 전에 스트림 설정이 동일한지 확인하십시오.
    스트림이 플레이어에서 멈춘 것으로 나타남 라이브 모듈이 입력 인코더에서 데이터 수신을 중지했을 가능성이 큽니다. 입력 스트림이 활성 상태인지 확인하고, 연결되어 있으며, 실시간 데이터보다 빠르게 전송되는지 확인합니다.
    재생이 고르지 않음 입력 스트림의 비트 전송률을 지원하기에 충분한 대역폭이 있는지 확인합니다. 고르지 않은 재생은 일반적으로 프로덕션 측에서 문제가 있거나 대역폭이 충분하지 않다는 표시입니다.
    빈 화면 대역폭이 충분한지 확인합니다. 연결이 끊어지는 경우 다시 연결하기 전에 설정이 변경되지 않았는지 확인하십시오.
    예상보다 긴 대기 시간 사용 가능한 대역폭을 확인하고 데이터가 라이브 모듈로 실시간으로 전송되는지 확인합니다. 스트림이 게시되려면 최대 10분 정도 걸릴 수 있습니다. 이 시간 동안 스트림이 캐시되고 대기 시간이 가장 길어집니다. 게시 후 대기 시간은 30-90초 사이에 안정화될 때까지 계속 감소합니다.
    계정에 따라 할당된 중복 스트림을 이미 사용했습니다. 계정 제한에 비해 중복 그룹이 있을 수 있습니다. 중복 스트림이 있는 이벤트가 완료되거나 취소되면 라이브 모듈에 더 이상 표시되지 않습니다. 그러나 이러한 이벤트는 Live API를 통해 나열되고 삭제할 수 있습니다. 라이브 API 참조: 중복 그룹.
     

    추가 도움말

    라이브 이벤트를 진행하는 데 추가 도움이 필요한 경우당사에 문의하실수 있습니다. 최대한 빠른 답변을 받을 수 있도록 브라이트코브 지원에서 문제를 해결하기 위해 필요한 사항의 목록을 아래에 제시했습니다.

    • 스트리밍에서 발생하는 특정 증상 예: 전혀 재생이 되지 않는다거나 잠깐씩 멈추거나 정지되는 증상
    • 과거 이 스트리밍이 올바르게 동작했는지 여부
    • 인코더에서 사용하는 진입점 URL
    • 사용 중인 인코딩 소프트웨어 및 하드웨어
    • 생중계를 게시한 플레이어에 연결된 URL
    • Video Cloud Studio에 있는 라이브 자산의 비디오 ID
    • 인코더에서 게시 지점 호스트로 경로를 추적한 결과
